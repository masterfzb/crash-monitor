# Crash Monitor v0.3.0 操作指南

> 适用: Windows 10/11 x64 | Python 3.8+ | 零外部依赖

---

## 一、这是什么

一个能抓到"电脑活着但核心服务悄悄死了"这类诡异故障的监控系统。

你的场景：开始菜单打不开、关机按钮废了、但桌面看起来正常。
普通工具抓不到原因（C 盘连日志都不写），crash-monitor 用以下手段解决：

- 每 5 秒检查 Windows 关键服务是否还在跑
- 所有日志/快照写入 **D 盘**，不碰 C 盘
- 异常瞬间自动拍"系统快照"——保留犯罪现场
- 浏览器打开仪表板，实时看系统健康状况
- **独立报告器** — 神授(Windows AI Agent)崩溃后失忆也能立刻查

```powershell
git clone https://github.com/masterfzb/crash-monitor.git D:\crash-monitor
cd D:\crash-monitor

## 三、配置说明

```yaml
windows_service_watchdog:
  enabled: true                 # 启用看门狗
  heartbeat_interval: 5         # 检查间隔（秒）
  log_dir: "D:\\crash-monitor\\logs"          # ⚠️ 必须非 C 盘
  status_dir: "D:\\crash-monitor\\status"
  snapshot_dir: "D:\\crash-monitor\\snapshots"

dashboard:
  enabled: true
  port: 19998

alerts:
  webhook:
    url: "https://qyapi.weixin.qq.com/cgi-bin/webhook/send?key=你的key"
    type: wecom
```

## 五、仪表板

`http://localhost:19998` — 深色主题，3 秒自动刷新：
- 🔧 **服务状态** — 绿=运行/黄=停止/红=挂了
- 📦 **进程存活** — 含 PID
- 🧪 **系统功能** — 关机可用？任务栏正常？
- 📋 **事件流** — ERROR 标红

## 六、神授失忆恢复协议 ⚠️

神授是 Windows 上的 AI Agent。系统崩溃重启后，她的会话是全新的——
她不记得刚才崩了什么。方先生或神谕问她，她必须立刻准确回复。

### 6.1 神授必须创建的文件

在神授的 OpenClaw workspace 下创建 `CRASH_RECOVERY.md`：

```markdown
# CRASH_RECOVERY.md — 神授崩溃恢复协议

## 我的身份
- 神授，Windows 10/11 x64，NVIDIA RTX 3090
- 和神谕(Mac Mini)是双神协作关系

## 环境速查
- OS: Windows x64（不是 Mac！不是 Linux！）
- 命令: PowerShell 优先，其次 cmd
- 关键服务: GPT-SoVITS，各种 Python API
- crash-monitor: D:\crash-monitor\

## 谨慎原则
- 不确定时先查，不盲改系统
- Windows 环境不要跑 bash/linux 命令
- 不要停掉 crash-monitor 进程
- 不要 taskkill /f explorer.exe
- 不要乱改注册表
- 神谕让你试命令时，先确认在 Windows 上合法

## 被问"刚才崩了什么"时

### 第一步：跑报告器
```powershell
python D:\crash-monitor\crash_report.py --live
```
直接复制输出回复。

### 第二步：报告器跑不了时手动查
```powershell
type D:\crash-monitor\status\heartbeat.txt
dir D:\crash-monitor\snapshots\ /O-D
powershell "Get-Content D:\crash-monitor\logs\watchdog_events.log -Tail 20"
sc query DcomLaunch
sc query RpcSs
tasklist | findstr explorer.exe
```

### 第三步：用这个模板回复
```
💥 崩溃报告:
- 系统已恢复运行 [✓/✗]
- 最后心跳: [时间]
- 异常服务: [列出]
- 死亡进程: [列出]
- 最近ERROR事件: [摘录]
```

## 危险命令清单（禁止）
- shutdown /r /f（除非明确要重启）
- taskkill /f explorer.exe
- del /f D:\crash-monitor\*（删快照=销毁证据）
- 任何注册表修改
- 停掉 crash-monitor
```

### 6.2 为什么这个文件能解决失忆

OpenClaw 每次启动会自动加载 workspace 下的文件到上下文。
`CRASH_RECOVERY.md` 在神授的工作目录里 → 她醒来就有这套协议。

### 6.3 现场报告器

```powershell
python D:\crash-monitor\crash_report.py         # 文本报告（适合聊天发送）
python D:\crash-monitor\crash_report.py --live  # 含实时系统检查
python D:\crash-monitor\crash_report.py --json  # JSON 格式
```

`crash_report.py` 是独立工具，不依赖 crash-monitor 运行。
神授重启后，无论 crash-monitor 启动没有，她都能跑它。

## 七、排查故障流程

### 当电脑异常时

1. 打开 `http://localhost:19998` 看哪些变红了
2. 或跑 `python D:\crash-monitor\crash_report.py --live`

### 异常后追溯

1. **翻事件日志**: `D:\crash-monitor\logs\watchdog_events.log` 从下往上找第一条 ERROR
2. **查故障快照**: `D:\crash-monitor\snapshots\` 找对应时间的 JSON
3. **对心跳**: `D:\crash-monitor\status\heartbeat.txt` 精度到秒

## 八、常见根因 & 修复

| 症状 | 可能原因 | 修复 |
|------|---------|------|
| 开始菜单死、关机废 | Fast Startup 损坏 | `powercfg /h off` |
| 间歇性服务挂 | svchost 承载过多 | 优化启动项 |
| 磁盘日志写不进 | NVMe 驱动/固件 | 更新芯片组驱动 |
| GPU 服务挂 | 驱动崩溃链式反应 | DDU + 重装驱动 |
| 开机后过一会挂 | 电源计划异常 | 重置电源计划 |

## 九、和 optimizerDuck 配合

optimizerDuck → 事前优化（减负载、关遥测、GPU 调优、关 Fast Startup）
crash-monitor → 事后兜底（崩了抓证据、告警、快照）

**建议顺序**: 先跑 optimizerDuck 关 Fast Startup → 再启 crash-monitor 常驻
